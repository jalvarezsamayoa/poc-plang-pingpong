# Project Overview: PingPong (P Language)

This project is a classic "Ping-Pong" demonstration implemented in the **P Programming Language**. P is a domain-specific language designed for modeling and formally verifying asynchronous distributed systems using state machines.

The project consists of two primary state machines:

- **Pinger:** Sends a `Ping` event to a `Ponger` and waits for a `Pong` event before repeating.
- **Ponger:** Listens for `Ping` events and responds with a `Pong` event.

The system is orchestrated by a `Main` machine that initializes both the `Pinger` and `Ponger` to start the interaction.

## Project Structure

- `pingpong.conf`: Central configuration file for the P compiler, defining source directories and output paths.
- `PSrc/`: Contains the core logic and state machine definitions.
  - `PingPong.p`: Defines the `Pinger` and `Ponger` machines and their transitions.
- `PSpecs/`: Contains system-level specifications and test machines.
  - `SystemTests.p`: Defines the `Main` entry point for simulation and verification.
- `PGenerated/`: Stores the artifacts generated by the P compiler, including C# source code and .NET binaries for model checking.

## Building and Running

The project uses the P compiler and checker (v3.0.4+). Ensure you have the [P toolchain](https://p-org.github.io/P/) and .NET 8.0 installed.

### Using the Makefile (Recommended)

The easiest way to build and verify the project is using the provided `Makefile`:

```bash
make compile  # Compiles the P project
make check    # Runs the P model checker
make clean    # Removes generated artifacts
```

### Manual Commands

To compile the P code into a .NET DLL for verification:

```bash
p compile -pp PingPong.pproj
```

### Run Model Checker

To run the P model checker and verify the system's correctness:

```bash
p check PGenerated/PChecker/net8.0/PingPong.dll
```

### Run Tests (Internal)

The generated output in `PGenerated/PChecker/` is a .NET project that can be run directly if needed, though `p check` is the standard way to verify P models.

## Development Conventions

- **State Machine Modeling:** All logic should be encapsulated within `machine` definitions using `state`, `entry`, and `on` handlers.
- **Event-Driven:** Communication between machines is strictly asynchronous and handled via `send` operations and `event` definitions.
- **Formal Verification:** Always run `p check` after logic changes to ensure no liveness or safety properties are violated.
- **Configuration:** Update `pingpong.conf` if new source or specification directories are added.
